CLI Neural Network Builder — Revised Project Plan
The Flow (Canonical Reference)
User Conversation → Scoping Questions → Data Requirements → Data Validation
        → Docker Environment Setup → Code Generation → Mock Run
                → Training → Inference Setup → Done
Every stage is a discrete, recoverable step. If any stage fails, the system reports clearly and gives the user an actionable path forward — it never silently moves on.

Architecture Overview
The tool has three distinct layers:
Orchestrator (CLI) — the user-facing layer. Manages conversation state, talks to Gemini, and drives the pipeline forward stage by stage.
Docker Runtime — an isolated container per project. All environment setup, code execution, mock runs, training, and inference live exclusively here. The host machine is never touched beyond Docker itself.
Gemini Brain — not a single call but a stateful agent across the pipeline. Each stage has its own prompt role: interviewer, analyst, validator, engineer, monitor, trainer, packager.

Stage-by-Stage Breakdown

Stage 0 — Project Init
The user runs nnb start and we create a project workspace.

Generate a unique project ID and a local .nnb/ directory to track all state
This directory persists conversation history, data snapshots, generated code, Docker configs, training logs, and the steering document — everything in one place
The steering document is auto-initialized here with defaults; the user can edit it before or after the conversation


Stage 1 — User Conversation
The user describes what they want to build in plain language.

Open-ended intake: nnb start drops into an interactive session
The user describes their project — domain, goal, constraints, prior work — with no required format
Gemini listens without interrupting, building an internal understanding of the project
Conversation history is saved after every turn so sessions can be resumed (nnb resume <project-id>)

What Gemini is doing internally: Categorizing the project (vision, NLP, tabular, time-series, multimodal, RL, other), estimating complexity, identifying ambiguities to resolve in Stage 2.

Stage 2 — Scoping Questions
Gemini surfaces the gaps it found and asks targeted follow-up questions.

Gemini generates a focused set of questions — not a survey, but a prioritized list of what it actually needs to make good decisions
Questions are asked conversationally, one topic at a time, not dumped all at once
Examples of what it probes: expected input/output shapes, latency vs. accuracy tradeoff preference, whether a pretrained base model is acceptable, compute budget, whether the model needs to be exportable (ONNX, TFLite, etc.)
Answers are folded back into the project's steering document automatically
At the end of this stage, Gemini produces a Project Specification — a structured summary of what was agreed — and asks the user to confirm before proceeding

Exit condition: User confirms the spec. This becomes the ground truth for all subsequent stages.

Stage 3 — Training Data Requirements
Gemini tells the user exactly what data it needs.

Based on the confirmed spec, Gemini generates a Data Requirements Report covering:

Format (image files, CSV, parquet, JSONL, audio, etc.)
Minimum and recommended dataset size
Required folder/label structure (e.g. data/train/class_a/, data/train/class_b/)
Any preprocessing it will handle automatically vs. what the user must do before handing it over
Gotchas: class imbalance thresholds, resolution requirements, encoding expectations


This report is saved to .nnb/data-requirements.md and printed to the terminal
The user is given a checklist: nnb data status shows which requirements are satisfied as they prepare their dataset


Stage 4 — Data Validation
Gemini takes a snapshot of the data and decides if it's ready.

User runs nnb data validate --path ./my-data
The tool takes a non-destructive sample of the data — not the full dataset, a representative slice — and passes it to Gemini
Gemini checks:

Structure matches the required format
Labels are present and consistent
No obviously corrupt files (truncated images, malformed rows, encoding errors)
Class distribution — flags severe imbalance with a recommendation
Rough size estimate to project training time


Output is a pass/warn/fail report. Warnings let you proceed; failures block the pipeline with a specific fix instruction
If data passes, a data manifest is written to .nnb/data-manifest.json — a fingerprint of what was validated. Docker will mount the validated path.


Stage 5 — Docker Environment Setup
A container is built and configured for this exact project.

A Dockerfile is generated by Gemini based on the confirmed spec:

Base image selected by framework (PyTorch CUDA, TensorFlow, JAX with TPU support, etc.)
Python version pinned
Dependencies derived from what the project will need — no generic bloated requirements, only what this model needs


The container is built: nnb env build
The validated data directory is mounted read-only into the container at a fixed path (/data)
The .nnb/workspace/ directory is mounted read-write as /workspace — all generated code and outputs land here on the host automatically
A health check runs inside the container after build: GPU/CPU availability, CUDA version, framework import, data mount accessible. Any failure is reported with the fix before we go further.

Key principle: The container is fully reproducible. The generated Dockerfile is committed alongside the project so anyone can recreate the exact environment.

Stage 6 — Code Generation + Mock Run
Gemini writes the code. Then the code has to prove it works before training starts.
Code Generation:

Gemini generates the full codebase inside the container at /workspace:

model.py — architecture definition
dataset.py — data loading and preprocessing pipeline
train.py — training loop with checkpointing
config.yaml — all hyperparameters externalized, never hardcoded
requirements.txt — redundant safety net


All code is written against the steering document's conventions (naming, docstrings, framework patterns)
Files appear on the host immediately via the volume mount

Mock Run:

Before any real training, a mock run executes inside the container: nnb mock-run
Mock run behavior:

Loads a tiny synthetic or sampled batch (2–4 samples)
Runs one forward pass, one backward pass, one optimizer step
Checks that shapes are correct end-to-end
Checks that loss is a valid number (not NaN, not Inf)
Checks GPU memory footprint against available VRAM


Gemini monitors the container logs in real time during the mock run
If anything fails, Gemini reads the traceback, generates a fix, patches the file, and re-runs automatically — up to a configurable retry limit (default: 3)
Mock run passing is a hard gate. Training cannot begin until the mock run is green.


Stage 7 — Training
Training runs inside the container. Gemini wakes when it's done.

nnb train kicks off training inside the container as a detached process
The terminal shows a live view: epoch progress, loss, metrics, ETA — streamed from container logs
The process is resilient: closing the terminal doesn't kill training. nnb attach <project-id> reattaches.
Checkpoints are saved to /workspace/checkpoints/ at configurable intervals, immediately visible on the host
Gemini is not active during training — it doesn't burn API calls watching a loss curve. It is woken up only when:

Training completes successfully
Training crashes (feeds the error to Gemini for diagnosis)
The user explicitly runs nnb check to ask Gemini to look at the current metrics


On completion, Gemini reviews the final training summary: final loss, validation metrics, any warning signs in the log. It writes a brief Training Report to .nnb/training-report.md.


Stage 8 — Inference Setup
Gemini packages the trained model for use.

Gemini generates inference artifacts inside the container:

inference.py — a clean, standalone inference script with a simple API (takes input, returns prediction)
server.py — optional FastAPI/Flask wrapper for REST inference (generated if the user indicated deployment intent during scoping)
Export to ONNX or TFLite if flagged in the steering document
inference-requirements.txt — minimal dependencies for inference only (often much lighter than training deps)


A quick inference smoke test runs automatically: loads the best checkpoint, runs one prediction on a sample input, verifies output shape and type
Final output summary is printed: model location, checkpoint path, how to run inference, file sizes

Done. The user has a trained model, a reproducible environment, and a ready-to-use inference interface.

State Machine
Every project lives in one of these states, tracked in .nnb/state.json:
INIT → SCOPING → SPEC_CONFIRMED → DATA_REQUIRED → DATA_VALIDATED
     → ENV_BUILDING → ENV_READY → MOCK_RUNNING → MOCK_PASSED
     → TRAINING → TRAINING_COMPLETE → INFERENCE_READY → DONE
Any command checks the current state first. nnb status always tells the user exactly where they are and what the next step is. No guessing.

Key Engineering Decisions
Container lifecycle — one container per project, not one per command. The container is built once in Stage 5 and reused through mock run, training, and inference. Rebuilding is explicit (nnb env rebuild).
Gemini as a staged agent, not a chatbot — each stage sends Gemini a purpose-built prompt with exactly the context it needs for that role. The Stage 2 Gemini is an interviewer; the Stage 6 Gemini is an engineer reading tracebacks. This produces much better outputs than a single long conversation.
Volume mounts as the integration boundary — the host never runs Python or touches training code. The container does all execution. Files flow between them exclusively through the mounted .nnb/workspace/ directory. This keeps the host clean and the environment perfectly isolated.
Steering document as a living artifact — it is updated automatically as the conversation progresses. By the time code generation starts, it is a complete, structured description of the project that Gemini uses as its ground truth for every decision.
